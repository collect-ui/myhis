### 0.3 必须修正的核心问题

1. **不能只保存号源主表**

   老源码 `addTemp` 不只是插 `PIX_OUTP_REG_MASTER`。完整链路至少包含：

   - 查挂号费别，补 `outp_type_code`、`outp_title_code`
   - 查专科，补 `outp_special_id`
   - 写 `pix_outp_reg_master`
   - 非分时写 `pix_outp_reg_pool`
   - 分时写 `pix_outp_reg_master_atime`
   - 分时生成并写 `pix_outp_reg_master_detail`
   - 同步写 `pix_outp_reg_schedule`
   - 分时同步写 `pix_outp_reg_schedule_atime`
   - 写 `pix_outp_reg_master_log`

2. **`outp_special_id` 是关键字段，不能漏**

   老源码在 `master.getOutpSpecialId() == null` 时，会按 `special_clinic_code` 查 `SPECIAL_CLINIC`，并要求 `delete_flag = '1'`。查不到直接报错。

   排班去重也用 `outp_special_id`，不是只用 `special_clinic_code`。低代码服务必须补齐这个字段。

3. **排班状态不能写字符串 `GENERATED`**

   老源码枚举注释显示 `schedule_status` 取值为：

   ```text
   0=已排班, 1=已列入执行计划, 2=已生成号源, 3=暂停, 4=作废
   ```

   新增临时号源生成排班时应写已生成号源枚举值，优先按老表值写 `"2"`，不要写文本 `GENERATED`。

4. **`modify_flag` 需要按老枚举确认**

   老源码新增时使用：

   ```java
   RegMasterEnum.RegMasterModifyFlag.NOT_UPDATED.typeValue
   ```

   不能只根据字面理解写死。当前文档早期版本把 `modify_flag = "1"` 解释为“未修改”，这个结论需要以源码枚举为准。

5. **分时号源必须生成详情表**

   分时模式不是只保存 `pix_outp_reg_master_atime`。老源码还会对每个分时时段按：

   - `appointment_limits` 生成可预约详情
   - `time_limits - appointment_limits` 生成不可预约详情
   - `internet_limits` 生成互联网详情

   写入 `pix_outp_reg_master_detail`。

6. **前端不能硬编码分时时段规则**

   当前页面 JSON 中通过长匿名表达式硬编码了时段配置，例如 `['2-9','10-12','13-19','20-33','34-1','34-33']`。这与老源码按专科配置读取时段不一致，也违反本项目“禁止前端 schema 匿名函数”的约束。

   正确方向：通过后端 service 按 `special_clinic_code + outp_duration_code` 返回分时点，前端只展示和提交结果。

7. **号池数组生成用低代码 handler 组合，不用前端循环**

   非分时号池不建议前端生成，也不建议写 Go 业务代码。推荐使用 `multi_arr(loop_max)` 直接生成号池行，再用 `update_array` 补字段、`combine_array` 合并三类号池。前提是这些 handler 已存在；不要为了这个流程临时改框架 Go 代码。

8. **CS 叫号不要简单改成本地插站位表**

   老源码是事务提交后调用：

   ```java
   csService.masterNotice(..., MasterEventEnum.ADD.getEventCode())
   ```

   `CsService` 会批量翻译后通过 Feign 调用 cs-server 的 `/cs/masterConsumer/masterNotice`。cs-server 内部可能包含站位类型、诊室排班、配置读取等规则。

   因此不能简单用 SQL 直写 `CS_SCHEDULE_STATION` 当作等价实现。除非已经完整核对 cs-server 逻辑和表结构，否则此步骤应标记为“待接 CS 通知能力”，不要伪实现。

9. **Redis 和 MQ/UMC 明确暂不实现**

   老源码会写 Redis 号池 key、设置 `PIX_REGISTER_INIT_FLAG`，并发送 UMC/MQ 号源变更消息。当前低代码框架没有直接等价能力时，文档必须明确：

   - Redis 不实现
   - MQ/UMC 不实现
   - 由此带来的补偿、缓存刷新、外部系统同步差异需要验收确认

### 0.4 雪花 ID 位数问题与 Mini 雪花方案

#### 问题

数据库表 `MASTER_ID NUMBER(12,0)` 最大存储 12 位数字（999,999,999,999）。

当前框架雪花 ID 是标准 64-bit 实现（`bwmarrin/snowflake`）：

```text
结构：1 bit 符号 + 41 bit 时间 + 10 bit 机器 + 12 bit 序列
最大值：9,223,372,036,854,775,807（19 位）
当前实际值：~599,751,917,568,004,096（18 位）
```

**结论：标准雪花 ID 是 18 位，NUMBER(12,0) 存不下。**

报错：`ORA-01438: value larger than specified precision allowed for this column`

#### Mini 雪花方案（40-bit，12 位以内）

将标准 64-bit 雪花压缩为 40-bit，产出 ≤ 999,999,999,999 的 ID。

**结构**：

```text
 位    位数    说明
[39]    1 bit  未用（始终 0，保证正数）
[38:8]  31 bit 时间戳（秒，自 epoch 以来）
[7:4]   4 bit  机器 ID（0-15）
[3:0]   4 bit  序列号（0-15）
合计    40 bit
```

**参数**：

| 参数 | 值 | 说明 |
|------|------|------|
| Epoch | 2024-01-01 00:00:00 UTC | Unix 秒 = 1704067200 |
| 时间位 | 31 bit | 2^31 秒 ≈ 68 年（可用至 ~2092 年） |
| 机器位 | 4 bit | 0-15，共 16 个节点 |
| 序列位 | 4 bit | 0-15，每秒每节点最多 16 个 ID |
| 最大值 | 2^40 - 1 | 1,099,511,627,775（12 位） |
| 精度 | 秒级 | 非毫秒 |

**与标准雪花对比**：

| | 标准雪花 | Mini 雪花 |
|---|---|---|
| 位数 | 64 bit | 40 bit |
| 最大值 | 9.2×10^18（19 位） | 1.1×10^12（12 位） |
| 时间精度 | 毫秒 | 秒 |
| 节点数 | 1024 | 16 |
| 每秒容量 | 4096/ms = 409.6 万/秒 | 16/秒 |
| 适用场景 | 分布式大数据 | 单体/小集群 HIS |

**实现要点**：

```go
// 伪代码
func MiniSnowID() int64 {
    now := time.Now().Unix() - epoch  // 秒级时间戳
    id := (now << 8) | (nodeID << 4) | step
    return id  // 最大 12 位
}
```

**注册方式**：在 `filters/all_register.go` 注册为 `snow_id_mini`，低代码通过 `{{snow_id_mini}}` 调用。

**兼容性**：现有 `{{snow_id}}` 保留不动，需要 12 位 ID 的地方改用 `{{snow_id_mini}}`。

### 0.6 修订后的低代码服务形态

#### 0.6.1 为什么不能继续用单表 `model_save`

当前目标不是“往号源主表插一行”，而是“新增一条可挂号、可排班、后续系统能识别的临时号源”。所以目标服务不应是：

```yaml
module: model_save
table: pix_outp_reg_master
```

单表 `model_save` 只会得到一条 `pix_outp_reg_master` 主表记录。它不会生成号池、不会生成分时详情、不会同步排班，也不会留下操作日志。最终表现是：列表上可能能看到一条号源，但挂号取不到可用号，叫号/预约/排班侧也不一定能识别。

目标服务应调整为一个编排型 service：

```yaml
module: empty
handler_params:
  # === Phase 1: 数据准备 ===
  - key: service2field   # 1. 查挂号费别（调用独立 SQL service）
  - key: service2field   # 2. 查专科 outp_special_id（调用独立 SQL service）
  - key: update_field    # 3. 补齐主表默认字段
  - key: update_field    # 4. 生成 master_id（Mini雪花ID，{{snow_id_mini}}）

  # === Phase 2: 写主表 ===
  - key: service2field   # 5. 写 pix_outp_reg_master（调用 model_save 类型的 service）

  # === Phase 3: 生成号池（非分时） ===
  - key: update_field    # 6. 计算 _no_appointment_count
  - key: multi_arr       # 7. 生成可预约号行（loop_max = appointment_limits）
  - key: update_array    # 8. 补充可预约号字段
  - key: multi_arr       # 9. 生成不可预约号行（loop_max = _no_appointment_count）
  - key: update_array    # 10. 补充不可预约号字段
  - key: multi_arr       # 11. 生成互联网号行（loop_max = internet_limits）
  - key: update_array    # 12. 补充互联网号字段
  - key: combine_array   # 13. 合并三类号池行
  - key: service2field   # 14. 写 pix_outp_reg_pool（调用 bulk_create 类型的 service）

  # === Phase 4: 生成分时详情（分时） ===
  - key: update_field    # 15. 补充 atime_rows 字段（分时主表）
  - key: service2field   # 16. 写 pix_outp_reg_master_atime（调用 bulk_create 类型的 service）
  - key: multi_arr       # 17. 生成分时详情行（loop_max × 3 类）
  - key: update_array    # 18. 补充分时详情字段
  - key: combine_array   # 19. 合并三类详情行
  - key: service2field   # 20. 写 pix_outp_reg_master_detail（调用 bulk_create 类型的 service）

  # === Phase 5: 同步排班 ===
  - key: service2field   # 21. 查旧排班（调用 SQL service）
  - key: service2field   # 22. 删除旧排班分时（调用 model_delete 类型的 service）
  - key: service2field   # 23. 删除旧排班（调用 model_delete 类型的 service）
  - key: service2field   # 24. 写新排班（调用 model_save 类型的 service）
  - key: service2field   # 25. 写排班分时（调用 bulk_create 类型的 service）

  # === Phase 6: 操作日志 ===
  - key: update_field    # 26. 拼接日志文本
  - key: service2field   # 27. 写 pix_outp_reg_master_log（调用 model_save 类型的 service）
```

**关键点**：`model_save`、`bulk_create`、`model_delete` 是 **module 级别**的操作，不能直接作为 `handler_params` 的 `key`。正确做法是为每个 DB 操作创建独立的 service（使用对应的 module），然后在编排 service 中通过 `service2field` 调用这些 service。

#### 0.6.2 入参、过程参数、出参的流转

前端提交的是业务入参，不是数据库最终字段。建议先按下面的参数池理解：

```text
前端入参
  area_code
  outp_date
  reside_dept_code
  special_clinic_code
  doctor_code
  outp_duration_code
  reg_fee_id
  registration_limits
  appointment_limits
  transfer_no
  internet_limits
  atime_flag
  registration_type
  average_time
  atime_rows
  operator

过程参数（handler_params 逐步补充）
  regFeeInfo              # 挂号费别对象
  specialClinicInfo       # 专科对象，至少包含 outp_special_id
  master_id               # 新号源主键（Mini雪花ID，{{snow_id_mini}}）
  outp_type_code          # 从 regFeeInfo 补
  outp_title_code         # 从 regFeeInfo 补
  current_limits          # = registration_limits
  appointment_current_limits # = appointment_limits
  poolRows                # 非分时号池行（每行 reg_pool_id 用 {{snow_id_mini}}）
  masterAtimeRows         # 分时主表行（每行 master_atime_id 用 {{snow_id_mini}}）
  detailRows              # 分时详情行（每行 master_detail_id 用 {{snow_id_mini}}）
  existingSchedule        # 查到的旧排班
  schedule_id             # 新排班主键（Mini雪花ID，{{snow_id_mini}}）
  scheduleAtimeRows       # 排班分时行（每行 schedule_atime_id 用 {{snow_id_mini}}）
  log_text                # 操作日志文本

数据库结果
  pix_outp_reg_master
  pix_outp_reg_pool
  pix_outp_reg_master_atime
  pix_outp_reg_master_detail
  pix_outp_reg_schedule
  pix_outp_reg_schedule_atime
  pix_outp_reg_master_log
```

关键点：`params` 是一条流水线。前面的 handler 写入 `params.regFeeInfo`、`params.master_id`、`params.poolRows`，后面的 handler 再读取这些字段继续处理。

#### 0.6.3 每一步具体做什么

1. 查挂号费别

   输入：`reg_fee_id`。

   输出到 params：`regFeeInfo`。

   用途：

   - `outp_type_code = regFeeInfo.outp_type_code`
   - `outp_title_code = regFeeInfo.outp_title_code`

   不能只写 `reg_fee_id`。老源码会从费别表补号别和职称/号标题编码。

2. 查专科 `outp_special_id`

   输入：`special_clinic_code`。

   输出到 params：`specialClinicInfo`。

   用途：

   - `outp_special_id = specialClinicInfo.outp_special_id`

   旧源码按 `special_clinic_code` 查专科，并过滤 `delete_flag = '1'`。查不到要失败。这个字段后续排班查重会用到，不能省。

3. 补齐主表字段

   输入：前端入参 + `regFeeInfo` + `specialClinicInfo`。

   输出到 params：

   ```text
   master_id
   outp_type_code
   outp_title_code
   outp_special_id
   current_limits = registration_limits
   appointment_current_limits = appointment_limits
   is_enable = "1"
   master_status = "0"
   current_no = 0
   upload_flag = "0"
   modify_flag = 老源码 NOT_UPDATED 枚举值
   ```

   注意：`modify_flag` 不要只凭文档写死，必须核对 `RegMasterEnum.RegMasterModifyFlag.NOT_UPDATED.typeValue`。

4. 写号源主表

   表：`pix_outp_reg_master`。

   这一步完成后，`master_id` 成为后续所有表的外键。

5. 非分时生成号池行

   条件：`atime_flag != "1"`。

   目标表：`pix_outp_reg_pool`。

   生成规则来自老源码 `regScheduleService.getRegPoolList(po)`，逻辑是：

   - 生成 `appointment_limits` 条可预约号，`appointment_flag = "1"`，`is_internet = "0"`
   - 生成 `registration_limits - appointment_limits` 条不可预约号，`appointment_flag = "0"`，`is_internet = "0"`
   - 生成 `internet_limits` 条互联网号，`appointment_flag = "1"`，`is_internet = "1"`

   每一行都要带：

   ```text
   reg_pool_id
   master_id
   outp_duration_code
   sort_no
   is_enable = "1"
   appointment_flag
   reg_flag = "0"
   is_internet
   time_stamp
   ```

   推荐用 `multi_arr` 的 `loop_max` 参数生成号池行，不让前端手写循环，也不需要 `service2field` 生成序号数组：

   ```text
   multi_arr(loop_max=appointment_limits) → _appointment_rows（可预约号）
     → update_array 补 sort_no=_seq / reg_pool_id / master_id / appointment_flag="1" / is_internet="0" 等

   multi_arr(loop_max=_no_appointment_count) → _no_appointment_rows（不可预约号）
     → update_array 补 sort_no=_seq / reg_pool_id / master_id / appointment_flag="0" / is_internet="0" 等

   multi_arr(loop_max=internet_limits) → _internet_rows（互联网号）
     → update_array 补 sort_no=_seq / reg_pool_id / master_id / appointment_flag="1" / is_internet="1" 等

   combine_array(_appointment_rows, _no_appointment_rows, _internet_rows) → _all_pool_rows
     → bulk_create pix_outp_reg_pool
   ```

   `multi_arr` 的 `loop_max` 模式不传 `right`，直接把模板复制 N 份，每份自动注入 `_seq`（1-based）。`update_array` 中 `sort_no = [_seq]` 直接引用。三类号分开生成是因为 `appointment_flag` 和 `is_internet` 值不同，`update_array` 不需要条件判断。

6. 写非分时号池

   条件：`atime_flag != "1"`。

   输入：`poolRows`。

   操作：`bulk_create pix_outp_reg_pool`。

7. 分时写主表分时行

   条件：`atime_flag == "1"`。

   输入：前端 `atime_rows`。

   目标表：`pix_outp_reg_master_atime`。

   每一行要补：

   ```text
   master_atime_id
   master_id
   sort_no
   time_status = "0"
   current_time_limits = time_limits
   current_appointment_limits = appointment_limits
   current_reserve_limits = reserve_limits
   current_internet_limits = internet_limits
   outp_date
   ```

   注意：旧源码批量新增时还处理夜班跨天 `outp_date`。如果当前低代码暂不实现跨天规则，文档和验收必须明确。

8. 分时生成详情行

   条件：`atime_flag == "1"`。

   输入：`masterAtimeRows`。

   目标表：`pix_outp_reg_master_detail`。

   对每个分时时段：

   - `appointment_limits` 生成可预约详情
   - `time_limits - appointment_limits` 生成不可预约详情
   - `internet_limits` 生成互联网详情

   每一行要带：

   ```text
   master_detail_id
   master_atime_id
   master_id
   time_quantum
   stage_begin_time
   stage_end_time
   outp_date
   sort_no
   is_enable = "1"
   appointment_flag
   reg_flag = "0"
   is_internet
   time_stamp
   ```

   这一步不能省。分时号源后续取号不是直接消费 `pix_outp_reg_master_atime`，而是消费详情行。

9. 查旧排班

   输入：

   ```text
   outp_special_id
   outp_duration_code
   outp_date
   doctor_code
   ```

   输出到 params：`existingSchedule`。

   老源码查重条件是同一个专科、同一天、同一个时段、同一个医生、启用且未删除。医生为空时要查 `doctor_code is null`，不能简单拼 `doctor_code = ''`。

10. 删除旧排班分时和旧排班

   为什么要删：

   老 HIS 规则是“临时号源生成后，要保证排班表里同一个医生/专科/日期/时段只有一条有效排班”。如果旧排班不删，新排班插入后会出现重复排班，预约平台、叫号系统、排班查询都可能显示两条。

   为什么先删分时再删主表：

   `pix_outp_reg_schedule_atime` 是排班分时子表，依赖 `schedule_id`。先删子表再删 `pix_outp_reg_schedule` 主表，避免外键或逻辑残留。

   注意：旧源码是物理删除旧排班和旧排班分时，不是把 `delete_flag` 改成 `'0'`。

11. 写新排班

   表：`pix_outp_reg_schedule`。

   字段来源大部分来自主号源：

   ```text
   schedule_id
   master_id
   area_code
   outp_special_id
   special_clinic_code
   reside_dept_code
   doctor_code
   outp_duration_code
   schedule_date = outp_date
   registration_limits
   appointment_limits
   reserve_limits = transfer_no
   internet_limits
   atime_flag
   registration_type
   average_time
   reg_fee_id
   outp_type_code
   outp_title_code

   delete_flag = "1"
   schedule_status = "2"
   ```

   `schedule_status = "2"` 表示已生成号源，不要写字符串 `GENERATED`。

12. 写排班分时

   条件：`atime_flag == "1"`。

   表：`pix_outp_reg_schedule_atime`。

   输入：号源分时行或前端 `atime_rows`。

   每行要补：

#### 0.6.4 Redis、MQ、CS 在低代码方案里的位置

老源码里这些步骤存在，但不是普通数据库表写入：

```text
非分时：
  saveRelPoolToRedis(master_id, regPoolList)

分时：
  RedisKeysUtil.lLeftPushAllStr(...)

全局刷新标记：
  redisUtil.set(PIX_REGISTER_INIT_FLAG, 当前时间)

CS 通知：
  事务提交后 csService.masterNotice(..., ADD)

UMC/MQ：
  umcMsgService.sendPixOutpRegMasterEdit(..., ADD)
```

修订后的低代码文档必须明确三件事：

1. Redis 不是“没有必要”，而是“当前低代码 handler 没有等价能力”。如果挂号取号依赖 Redis 队列，单纯写数据库仍然可能不可挂号。
2. Redis 需要的不是一个字段，而是多组队列：
   - 非分时可预约：`noTimeSharingY(master_id)`
   - 非分时不可预约：`noTimeSharingN(master_id)`
   - 分时可预约：`timeSharingY(master_id:time_section)`
   - 分时不可预约：`timeSharingN(master_id:time_section)`
   - 互联网分时：`internetTimeSharing(master_id:time_section)`
   - 互联网非分时：`internetNoTimeSharing(master_id)`
   - 全局标记：`PIX_REGISTER_INIT_FLAG`
3. 如果第一阶段不实现 Redis/MQ/CS，文档和验收必须写清楚影响范围：数据库记录可生成，但实时取号、缓存刷新、外部叫号和第三方同步不保证等价。

建议把 Redis/MQ/CS 放在第二阶段，不要混进第一阶段的纯低代码数据库闭环里伪实现。后续如果必须实现 Redis，应该新增明确的 Redis handler/plugin 或服务端能力，而不是用 SQL 代替。

#### 0.6.5 推荐的号池数组生成方式（`multi_arr` + `loop_max`）

非分时号池建议不要在前端 JSON 里手写循环，也不需要 `service2field` 生成序号数组。

`multi_arr` 已支持 `loop_max` 参数：不传 `right`，直接把 `field` 数组的每个元素复制 `loop_max` 份，每份自动注入 `_seq`（1-based）。

然后用 handler 组合生成完整号池行：

```text
1. update_field 定义 _pool_template
2. update_field 计算 _no_appointment_count = registration_limits - appointment_limits
3. multi_arr(loop_max=appointment_limits) → _appointment_rows（可预约号）
4. update_array(_appointment_rows) 补：
   - sort_no = _seq（由 multi_arr 自动注入，1-based）
   - reg_pool_id = snow_id
   - master_id
   - outp_duration_code
   - appointment_flag = "1"
   - is_internet = "0"
   - reg_flag = "0"
   - is_enable = "1"
   - time_stamp = unix_time(0)
5. multi_arr(loop_max=_no_appointment_count) → _no_appointment_rows（不可预约号）
6. update_array(_no_appointment_rows) 补：
   - sort_no = _seq
   - reg_pool_id = snow_id
   - master_id
   - outp_duration_code
   - appointment_flag = "0"
   - is_internet = "0"
   - reg_flag = "0"
   - is_enable = "1"
   - time_stamp = unix_time(0)
7. multi_arr(loop_max=internet_limits) → _internet_rows（互联网号）
8. update_array(_internet_rows) 补：
   - sort_no = _seq
   - reg_pool_id = snow_id
   - master_id
   - outp_duration_code
   - appointment_flag = "1"
   - is_internet = "1"
   - reg_flag = "0"
   - is_enable = "1"
   - time_stamp = unix_time(0)
9. combine_array(_appointment_rows, _no_appointment_rows, _internet_rows) → _all_pool_rows
10. bulk_create pix_outp_reg_pool data=[_all_pool_rows]
```

三类号分开生成的原因：`appointment_flag` 和 `is_internet` 值不同，分开生成后各组字段固定，`update_array` 不需要条件判断。

分时详情建议新增 SQL service：

```text
him.pix_outp_reg_master_gen_detail_rows
```

输入：

```text
masterAtimeRows 或 master_id
```

输出：完整 `pix_outp_reg_master_detail` 行数组。

分时详情比非分时号池复杂，因为它要按每个分时时段分别展开三类号。可以继续设计类似的 `multi_arr + update_array + combine_array` 链路，但需要先确认 handler 是否支持“按数组中每个时段分别生成 N 行”。如果不支持，分时详情用 SQL service 会更清楚。

具体字段、枚举值、SQL 仍需按老源码和 Oracle 表结构逐项核对。

## 一、整体背景

### 1.1 产品背景

**医院挂号系统**的核心能力是"放号"——医生排好班后，系统为他产生可挂的号源。患者来挂号时，从号源里取一个号。

号源有三个关键属性：
1. **哪个医生**的号（张三还是李四）
2. **什么时间**的号（周一上午还是下午）
3. **有多少个**号（限号50还是100）

一条号源包含：医生 + 专科 + 日期 + 时段 + 总限号数 + 预约限号数。

### 1.2 技术背景

老系统（pix-server）新增一条临时号源不是单表保存，而是一条完整业务链路：查费别→查专科→补字段→写主表→生号池或分时详情→同步排班→写日志→写 Redis 队列/刷新标记→事务提交后通知分诊叫号系统→发送 UMC/MQ 变更消息。

新系统用低代码框架迁移时，需要把这条链路拆成三类能力：

1. **第一阶段必须实现的数据库闭环**
   - 使用现有 handler：`service2field`、`update_field`、`model_save`、`bulk_create`、`model_delete`
   - 不新建 Go plugin
   - 覆盖主表、号池/分时、排班、日志
2. **第二阶段必须评估的运行时能力**
   - Redis 队列和 `PIX_REGISTER_INIT_FLAG`：老系统用于取号队列和缓存刷新，当前低代码没有等价 handler，不能假装已实现
   - 分诊叫号（cs-server）：老系统是事务提交后调用 `/cs/masterConsumer/masterNotice`，应优先走真实通知能力；不要简单本地 SQL 直写站位表
3. **可暂缓但必须标注影响的外部同步**
   - UMC/MQ：当前低代码无 RocketMQ/UMC 客户端时暂不实现，但要在验收里说明外部平台不会收到号源变更消息

本流程图按“数据库闭环 + 外部同步边界”重新梳理：

### 2.1 查挂号费别

#### 产品经理说

**背景**：用户在新增排班时，需要选择一个"挂号费别"——比如普通号30元、专家号100元、急诊号50元。这个费别决定了患者挂号时付多少钱。

**目的**：前端只保存了费别的ID（一串数字），但号源记录需要同时保存"号别编码"（普通号是1、专家号是3）。号别编码不在前端，在费别配置表里。

**解决什么问题**：如果不查费别，号源记录就不知道是"普通号"还是"专家号"，后续统计和报表会出错。

**做了什么**：用户选了"普通号（30元）"→系统拿到费别ID→去费别表查出→"哦，普通号对应的号别编码是1"→把1写回到号源记录里。

**一句话总结**：前端选费别 → 后端查出费别的编码 → 回填号码源。

#### 技术经理说

**涉及的表**：

### 2.2 查专科并补全号源主表字段

#### 产品经理说

**背景**：号源主表有十几个字段，一部分是前端填的（如医生、日期、限号数），另一部分需要系统自动计算或查表补齐。

**目的**：
1. `current_limits`（当前可用数）= 总限号数——因为刚建好的号源还没人挂，全部可用
2. `appointment_current_limits`（预约当前可用）= 预约限号数——同理
3. `outp_type_code`（号别编码）← 从第2.1步查出的费别信息里取
4. `outp_title_code`（号标题/职称编码）← 从第2.1步查出的费别信息里取
5. `outp_special_id`（专科内部主键）← 按 `special_clinic_code` 查专科表

**解决什么问题**：如果不补这些字段，数据库会有空值；如果 `current_limits` 不初始化为总限号数，系统会认为这个号源一个号都没有；如果 `outp_special_id` 为空，后续排班去重和排班同步会对不上老系统规则。

**做了什么**：按专科编码查专科主键；把"总限号100"复制一份给"当前可用数"；把"预约限号60"复制给"预约当前可用"；把费别里的号别编码和号标题编码填过来；补启用、正常、未上传、未修改等默认标记。

**一句话总结**：把前端传来的业务字段，补成老 HIS 主表真正需要的完整字段。

#### 技术经理说

**这个步骤是做什么的**：
这一步由两个动作组成：

1. `service2field` 查专科表，把 `special_clinic_code` 转成 `outp_special_id`。
2. `update_field` 修改或补充 params 中的字段。它不做数据库操作，只改内存中的数据。

在这里，我们需要在把数据写入号源主表之前，把一些"需要计算或查表才能得到"的字段提前准备好。Java 代码里用 `po.setXxx()` 做的事，这里用 `update_field` 做。

**要解决的问题**：
1. `current_limits`（当前可用数）—— 如果这个字段是空的，系统会认为这个号源一个号都没有，患者无法挂号。所以必须在 INSERT 之前把它初始化成总限号数。
2. `outp_type_code`（号别编码）—— 前端只传了费别ID，没传号别编码。如果漏填，号源记录就不知道是"普通号"还是"专家号"，后续统计报表会出错。
3. `outp_title_code`（号标题编码）—— 老源码和旧表都回填这个字段，不能只补 `outp_type_code`。
4. `outp_special_id`（专科内部主键）—— 老源码按它查重排班。只传 `special_clinic_code` 不够。
5. `upload_flag`、`modify_flag`、`master_status`、`current_no`—— 业务默认值。漏掉会导致查询、调整、上传状态和后续逻辑不一致。

**数据流向**：
```
执行前 params:
  registration_limits: 100     ← 前端传的（总限号）
  appointment_limits: 60       ← 前端传的（预约限号）
  regFeeInfo.outp_type_code: "1"  ← 第2.1步查到的（号别编码）
  regFeeInfo.outp_title_code: "01" ← 第2.1步查到的（号标题编码）
  special_clinic_code: "10500114"  ← 前端传的（专科编码）
  outp_special_id: null        ← 需要查专科表补齐
  current_limits: null         ← 需要补
  upload_flag: null            ← 需要补

查专科后 params:
  specialClinicInfo.outp_special_id: 12345

执行 update_field 后 params:
  outp_special_id: 12345
  current_limits: 100           ← 从 registration_limits 复制
  appointment_current_limits: 60 ← 从 appointment_limits 复制
  outp_type_code: "1"           ← 从 regFeeInfo 取
  outp_title_code: "01"         ← 从 regFeeInfo 取
```

### 2.4 生成号池（非分时模式）

#### 产品经理说

**背景**：号源需要有"库存"。限号100个，意味着有100个号可以挂。系统需要为这100个号在号池表里创建100条记录，每条记录代表一个"号"——患者挂号时从号池里取一个号。

但这100个号不能全部放出去预约，要分三类：
1. **可预约号**（比如60个）：患者可以在线或电话提前预约的号
2. **不可预约号**（比如40个）：只能当天来医院现场挂的号
3. **互联网号**（比如10个）：通过互联网渠道（如医院APP、公众号）挂的号

注意：互联网号是从可预约号里单独划出来的，所以实际：
- 可预约号 = 预约限号数（如60）
- 不可预约号 = 总限号 - 预约限号（如100-60=40）
- 互联网号 = 互联网限号（如10，这10个从可预约的60个里出）

**目的**：为每个号创建一条独立的记录，每条记录有唯一的ID、排序号，以及标记（是否可预约、是否互联网渠道）。

**解决什么问题**：如果没有号池，患者挂号时就没号可取。而且不分可预约/不可预约的话，预约平台会把所有号都放出去，当天来的患者就没号挂了。

**做了什么**：根据限号数先生成序号数组，再用 `multi_arr` 批量复制号池模板，用 `update_array` 给每行补字段，最后用 `combine_array` 把普通号池和互联网号池合并后写入号池表。每条记录都会标记好"可预约/不可预约/互联网"。

**一句话总结**：限号100个→号池表里生成100条记录，每条一个号，分成可预约/不可预约/互联网三类。

#### 技术经理说

**涉及的表**：
- `PIX_OUTP_REG_POOL`（号池表）：INSERT N 条记录

**数据流向**：
```
params.registration_limits = 100
params.appointment_limits = 60
params.internet_limits = 10
params._no_appointment_count = registration_limits - appointment_limits = 40
        │
        ▼
multi_arr(loop_max=appointment_limits=60) → _appointment_rows（可预约号）
        │
        ▼
update_array 补 sort_no=_seq / reg_pool_id / master_id / appointment_flag="1" / is_internet="0" 等
        │
        ▼
multi_arr(loop_max=_no_appointment_count=40) → _no_appointment_rows（不可预约号）
        │
        ▼
update_array 补 sort_no=_seq / reg_pool_id / master_id / appointment_flag="0" / is_internet="0" 等
        │
        ▼
multi_arr(loop_max=internet_limits=10) → _internet_rows（互联网号）
        │
        ▼
update_array 补 sort_no=_seq / reg_pool_id / master_id / appointment_flag="1" / is_internet="1" 等
        │
        ▼
combine_array(_appointment_rows, _no_appointment_rows, _internet_rows) → _all_pool_rows
        │
        ▼
bulk_create → INSERT ALL 到 pix_outp_reg_pool
```

```java
// 注: yList/nList/hlwList 用于后面写Redis，低代码不实现Redis
```

| `appointmentFlag` 分界 | `update_array` | 三类号分开生成，各自设置 `appointment_flag` 和 `is_internet` |

| `regPoolService.saveBatch(list)` | `bulk_create pix_outp_reg_pool` | 一次性批量写入 |

**为什么这么实现**：
- 低代码框架不适合在前端写循环，所以用 `multi_arr(loop_max)` 直接按数量生成行。
- `multi_arr(loop_max)` 不传 `right`，直接把模板复制 N 份，每份自动注入 `_seq`（1-based）。
- 用 `update_array` 负责逐行补业务字段，`sort_no = [_seq]` 直接引用。
- 三类号（可预约、不可预约、互联网）分开生成是因为 `appointment_flag` 和 `is_internet` 值不同，`update_array` 不需要条件判断。
- 最后用 `combine_array` 合并，`bulk_create` 一次性写入。

### 2.5 保存排班

#### 产品经理说

**背景**：每次新增号源，都需要同步更新排班表。排班表是给叫号系统、预约平台等外部系统使用的——它们不直接读号源表，而是读排班表。

**目的**：在排班表里创建一条记录，记录"医生X在某天某时段在Y专科出诊"。

**关键规则**：同一个医生+同一个专科+同一天+同一个时段，不能有两条排班。如果已经存在了（比如之前已经建过号源又删了），必须先删掉旧的再创建新的。

**解决什么问题**：如果不删旧排班，同一个时段就会有两条排班，外部系统会显示两次，患者会看到重复的号。

**做了什么**：先查有没有旧排班→有就删→插新排班→如果是分时模式还要插排班分时。

**一句话总结**：保证每个医生每天每个时段只有一条排班，有旧的先删再插。

#### 技术经理说

**涉及的表**：
- `PIX_OUTP_REG_SCHEDULE`（排班表）：INSERT / DELETE
- `PIX_OUTP_REG_SCHEDULE_ATIME`（排班分时表）：INSERT / DELETE

**数据流向**：
```
查旧排班
  │
  ├─ 有旧排班 → 删排班分时 → 删排班主表
  │
  └─ 无旧排班 → 直接下一步
  │
  ▼
插新排班 → 插排班分时（分时才插）
```

**原始 Java 代码**（`PixOutpRegScheduleService.java:2494-2530`）：

```java
public void addForTempMaster(PixOutpRegMasterPO po, List<PixOutpRegMasterAtime> atimes) {
    // 校验是否有排班，如果有就删掉
    PixOutpRegSchedulePO schedulePO = RegScheduleFactory.createPixOutpRegSchedulePO(po);
    LambdaQueryWrapper<PixOutpRegSchedulePO> queryWrapper = new LambdaQueryWrapper<>();
    queryWrapper.eq(PixOutpRegSchedulePO::getOutpSpecialId, schedulePO.getOutpSpecialId())
        .eq(PixOutpRegSchedulePO::getOutpDurationCode, schedulePO.getOutpDurationCode())
        .eq(PixOutpRegSchedulePO::getScheduleDate, schedulePO.getScheduleDate())
        .eq(PixOutpRegSchedulePO::getDoctorCode, schedulePO.getDoctorCode())
        .eq(PixOutpRegSchedulePO::getIsEnable, CommonEnum.IsEnable.ENABLE.typeValue)
        .eq(PixOutpRegSchedulePO::getDeleteFlag, CommonEnum.DeleteFlag.UNDELETE.typeValue);
    if (StringUtils.isEmpty(schedulePO.getDoctorCode())) {
        queryWrapper.isNull(PixOutpRegSchedulePO::getDoctorCode);
    } else {
        queryWrapper.eq(PixOutpRegSchedulePO::getDoctorCode, schedulePO.getDoctorCode());
    }
    List<PixOutpRegSchedulePO> pixOutpRegSchedulePO = list(queryWrapper);
    if (!CollectionUtils.isEmpty(pixOutpRegSchedulePO)) {
        List<Long> scheduleIds = pixOutpRegSchedulePO.stream().map(PixOutpRegSchedulePO::getScheduleId).collect(Collectors.toList());
        pixOutpRegScheduleAtimeService.deleteByScheduleIds(scheduleIds);
        pixOutpRegScheduleMapper.deleteBatchIds(scheduleIds);

    save(schedulePO);
    if (RegScheduleEnum.AtimeFlag.ENABLE_ATIME.typeValue.equals(po.getAtimeFlag())) {
        List<PixOutpRegScheduleAtimePO> regScheduleAtimePOS = BeanConvertUtil.convertListBean(atimes, PixOutpRegScheduleAtimePO.class);
        regScheduleAtimePOS.forEach(atimePO -> {
            atimePO.setScheduleId(schedulePO.getScheduleId());
        });
        pixOutpRegScheduleAtimeService.saveBatch(regScheduleAtimePOS);
    }
}
```

### 2.7 平均时长日志（可按平均看诊时间模式）

### 2.8 推送分诊叫号系统

#### 产品经理说

**背景**：患者挂完号后，需要在诊室门口的屏幕上看到自己的排队位置——前面还有几个人、轮到几号了。这个功能叫"分诊叫号"。

分诊叫号系统（cs-server）自己维护一个"站位表"（`CS_SCHEDULE_STATION`），记录每个医生当前有哪些患者在排队。当 HIS 新增了号源，需要把这个号源信息告诉分诊叫号系统，让它创建对应的站位记录。

**目的**：新增号源后，通知分诊叫号系统在站位表里创建站位记录，这样患者挂号后才能在叫号屏幕上显示。

**解决什么问题**：如果不通知分诊叫号系统，患者挂了号但叫号屏上看不到他，护士不知道下一个该叫谁，整个门诊流程就断了。

**做了什么**：号源数据库事务保存成功后，通知 cs-server 的 `masterNotice` 接口，把号源数据和事件类型 `ADD` 传过去。cs-server 收到后按自己的规则创建站位记录、处理诊室排班等。

**一句话总结**：新增号源 → 推送给分诊叫号系统 → 叫号屏上才能看到站位信息。

#### 技术经理说

**这个步骤是做什么的**：
分诊叫号系统在 cs-server 中维护站位、诊室排班、叫号显示等数据。当 HIS 新增号源时，HIS 的职责不是直接拼站位表 SQL，而是把“新增号源事件”通知给 cs-server，让 cs-server 用自己的业务规则处理。

老系统真实调用链是：

```text

  → 数据库事务内保存号源、号池/分时、排班、日志
  → 事务提交后
  → CsService.masterNotice(masterList, MasterEventEnum.ADD)
  → CsFeign.masterNotice(...)
  → POST /cs/masterConsumer/masterNotice
  → cs-server 自己查询号源详情并生成站位/诊室排班
```

```java
    Map<Long, MasterDTO> masterMap =  masterDTOList.stream().collect(Collectors.toMap(MasterDTO::getMasterId, value -> value, (key1, key2) -> key2));
    Map<String, List<MasterDTO>> collect = masterDTOList.stream().collect(Collectors.groupingBy(item -> item.getResideDeptCode() + StrPool.DASH + item.getOutpDurationCode()));
    Map<String, String> durationTimeRangeMap = csScheduleStationService.getOutpDurationTimeRangeMap(collect.keySet());

        if (CollUtil.isNotEmpty(subMasterIds)) {
            bathchHandleByMasterIds(specialClinicMap, masterMap, durationTimeRangeMap, subMasterIds);
        }
    }
}

    List<RegMasterDetail> details = regMasterDetailService.findByMasterIds(masterIds);
    if (CollUtil.isNotEmpty(details)) {

                    } else {

                        roomScheduleId = csRoomSchedulePO.getId();

                    }
                }
                Long edId = csScheduleStationPool.getNext();
                Long firstId = csScheduleStationPool.getNext();
                Long revistId = csScheduleStationPool.getNext();
                RegMasterDetail regMasterDetail = value.get(i);
                SpecialClinicVO specialClinicVO = specialClinicMap.get(masterDTO.getSpecialClinicCode());
                CsScheduleStation jz = CsScheduleStationAssembler.generateScheduleStation(masterDTO, regMasterDetail, specialClinicVO, durationTimeRangeMap);

    }
```

### 2.9 批量数据库写入与异常回滚

#### 产品经理说

**背景**：前面所有步骤都在内存中准备好了数据（号源主表、号池、分时、详情、日志等），最后需要一次性写入数据库。如果在写入过程中出错了，已经写了一半的数据必须全部撤销，否则会出现"号源主表有记录但号池没有"的不一致状态。

**目的**：把所有数据在一个事务中批量写入，出错了全部回滚。

**解决什么问题**：保证数据一致性——要么全部写入成功，要么全部不写。

**做了什么**：批量 INSERT 5 张表，捕获异常后清理 Redis Key，然后抛出业务异常回滚事务。

**一句话总结**：5 张表的数据一次性写入，失败了全部回滚。

#### 技术经理说

**这个步骤是做什么的**：
把前面构建的所有 PO 列表批量写入数据库。如果任何一步失败，清理 Redis 后回滚整个事务。

**原始 Java 代码**（`PixOutpRegMasterService.java:1558-1577`，批量新增 `batchAdd` 路径的统一批量写入示例）：
```java
try {
    // ===== 批量写入 5 张表（在同一事务中） =====
    pixOutpRegMasterService.saveBatch(masterPOS);       // 号源主表
    regPoolService.saveBatch(regPoolPOS);                // 号池表（非分时）
    pixOutpRegMasterAtimeService.saveBatch(atimePOList); // 分时主表
    regMasterDetailService.saveBatch(detailPOS);         // 分时详情
    pixOutpRegMasterLogService.saveBatch(logPOS);        // 操作日志

} catch (Exception e) {
    // ===== 异常回滚：清理 Redis Key =====
    List<String> keys = Lists.newArrayList();
    masterIds.forEach(masterId -> {
        keys.add(RedisKeyName.noTimeSharingY(masterId));    // 非分时可预约
        keys.add(RedisKeyName.noTimeSharingN(masterId));    // 非分时不可预约
        keys.add(RedisKeyName.timeSharingY(masterId));      // 分时可预约
        keys.add(RedisKeyName.timeSharingN(masterId));      // 分时不可预约
        keys.add(RedisKeyName.internetTimeSharing(masterId));   // 互联网分时
        keys.add(RedisKeyName.internetNoTimeSharing(masterId)); // 互联网非分时
    });
    redisUtil.delete(keys);  // 清理已写入的 Redis Key
    throw new BizException(PixExceptionCode.REG_MASTER_TEMP_ERROR, e);
}
```

## 四、所有 handler_params 执行顺序

> **注意**：`model_save`、`bulk_create`、`model_delete` 是 module 级别操作，不能直接作为 handler_params 的 key。
> 正确做法是为每个 DB 操作创建独立 service，然后通过 `service2field` 调用。

```
=== Phase 1: 数据准备 ===
① service2field  查费别（调用 SQL service）
② arr2obj        费别数组转对象
③ update_field   补全字段
④ update_field   生成 master_id（Mini雪花ID，{{snow_id_mini}}）

=== Phase 2: 写主表 ===
⑤ service2field  写号源主表（调用 model_save 类型的 service）

=== Phase 3: 生成号池（非分时） ===
⑥ update_field   计算 _no_appointment_count = registration_limits - appointment_limits
⑦ multi_arr      生成可预约号行（loop_max = appointment_limits）
⑧ update_array   补 sort_no/reg_pool_id/master_id/appointment_flag/is_internet
⑨ multi_arr      生成不可预约号行（loop_max = _no_appointment_count）
⑩ update_array   补 sort_no/reg_pool_id/master_id/appointment_flag/is_internet
⑪ multi_arr      生成互联网号行（loop_max = internet_limits）
⑫ update_array   补 sort_no/reg_pool_id/master_id/appointment_flag/is_internet
⑬ combine_array  合并三类号池行
⑭ service2field  写号池（调用 bulk_create 类型的 service）

=== Phase 4: 生成分时详情（分时） ===
⑮ update_field   补充 atime_rows 字段
⑯ service2field  写分时主表（调用 bulk_create 类型的 service）
⑰ multi_arr      生成分时可预约详情行（loop_max = appointment_limits）
⑱ update_array   补 master_detail_id/master_atime_id/time_quantum 等
⑲ multi_arr      生成分时不可预约详情行（loop_max = _no_appointment_count）
⑳ update_array   补字段
㉑ multi_arr      生成分时互联网详情行（loop_max = internet_limits）
㉒ update_array   补字段
㉓ combine_array  合并三类详情行
㉔ service2field  写分时详情（调用 bulk_create 类型的 service）

=== Phase 5: 同步排班 ===
㉕ service2field  查旧排班（调用 SQL service）
㉖ service2field  删旧排班分时（调用 model_delete 类型的 service）
㉗ service2field  删旧排班（调用 model_delete 类型的 service）
㉘ service2field  插新排班（调用 model_save 类型的 service）
㉙ service2field  插排班分时（调用 bulk_create 类型的 service）

=== Phase 6: 操作日志 ===
㉚ update_field   拼接日志文本
㉛ service2field  写操作日志（调用 model_save 类型的 service）
```

### 6.4 验收标准与 SQL 验证

#### Step 1：查号源主表

```sql
-- 应有 1 条记录
SELECT master_id, area_code, outp_date, special_clinic_code, doctor_code,
       outp_duration_code, registration_limits, appointment_limits,
       atime_flag, outp_type_code, outp_title_code, outp_special_id,
       is_enable, master_status, current_no
FROM pix_outp_reg_master
WHERE outp_date = '2026-06-14'
  AND special_clinic_code = '22222'
  AND doctor_code = 'hujun'
ORDER BY master_id DESC
FETCH FIRST 1 ROWS ONLY;
```

**验证点**：
- `master_id` 为 Mini 雪花 ID（≤12 位）
- `outp_type_code` = '1'（从 reg_fee_id 查到）
- `outp_special_id` 不为空（从 special_clinic_code 查到）
- `is_enable` = '1'
- `master_status` = '0'

#### Step 2：查号池表（非分时）

```sql
-- 非分时：应有 registration_limits 条记录
-- 可预约号：appointment_limits 条（appointment_flag='1', is_internet='0'）
-- 不可预约号：registration_limits - appointment_limits 条（appointment_flag='0', is_internet='0'）
SELECT appointment_flag, is_internet, COUNT(*)
FROM pix_outp_reg_pool
WHERE master_id = (SELECT master_id FROM pix_outp_reg_master
                   WHERE outp_date = '2026-06-14' AND special_clinic_code = '22222'
                   ORDER BY master_id DESC FETCH FIRST 1 ROWS ONLY)
GROUP BY appointment_flag, is_internet;
```

**验证点**：
- 非分时（atime_flag='0'）：应有 100 条记录（60 可预约 + 40 不可预约）
- 分时（atime_flag='1'）：此表应为空

#### Step 3：查分时主表（分时）

```sql
-- 分时：应有 atime_rows 数量的记录
SELECT master_atime_id, master_id, time_quantum, sort_no,
       time_status, current_time_limits, current_appointment_limits
FROM pix_outp_reg_master_atime
WHERE master_id = (SELECT master_id FROM pix_outp_reg_master
                   WHERE outp_date = '2026-06-14' AND special_clinic_code = '22222'
                   ORDER BY master_id DESC FETCH FIRST 1 ROWS ONLY)
ORDER BY sort_no;
```

**验证点**：
- 分时（atime_flag='1'）：应有 48 条记录（与 atime_rows 数量一致）
- 非分时（atime_flag='0'）：此表应为空
- `master_atime_id` 为 Mini 雪花 ID（≤12 位）

#### Step 4：查分时详情表（分时）

```sql
-- 分时：每个时段应有 appointment_limits + (time_limits - appointment_limits) 条记录
SELECT master_atime_id, time_quantum, appointment_flag, is_internet, COUNT(*)
FROM pix_outp_reg_master_detail
WHERE master_id = (SELECT master_id FROM pix_outp_reg_master
                   WHERE outp_date = '2026-06-14' AND special_clinic_code = '22222'
                   ORDER BY master_id DESC FETCH FIRST 1 ROWS ONLY)
GROUP BY master_atime_id, time_quantum, appointment_flag, is_internet
ORDER BY master_atime_id, appointment_flag;
```

**验证点**：
- 分时（atime_flag='1'）：每个时段应有详情记录
- `master_detail_id` 为 Mini 雪花 ID（≤12 位）

#### Step 5：查排班表

```sql
-- 应有 1 条排班记录
SELECT schedule_id, master_id, outp_special_id, special_clinic_code,
       doctor_code, outp_duration_code, schedule_date,
       registration_limits, appointment_limits, schedule_status, is_enable
FROM pix_outp_reg_schedule
WHERE master_id = (SELECT master_id FROM pix_outp_reg_master
                   WHERE outp_date = '2026-06-14' AND special_clinic_code = '22222'
                   ORDER BY master_id DESC FETCH FIRST 1 ROWS ONLY);
```

**验证点**：
- `schedule_id` 为 Mini 雪花 ID（≤12 位）
- `schedule_status` = '2'（已生成号源）
- `is_enable` = '1'

#### Step 6：查排班分时表（分时）

```sql
-- 分时：应有与 atime_rows 数量一致的排班分时记录
SELECT schedule_atime_id, schedule_id, master_id, time_quantum, sort_no
FROM pix_outp_reg_schedule_atime
WHERE master_id = (SELECT master_id FROM pix_outp_reg_master
                   WHERE outp_date = '2026-06-14' AND special_clinic_code = '22222'
                   ORDER BY master_id DESC FETCH FIRST 1 ROWS ONLY)
ORDER BY sort_no;
```

**验证点**：
- 分时（atime_flag='1'）：应有 48 条记录
- `schedule_atime_id` 为 Mini 雪花 ID（≤12 位）

#### Step 7：查操作日志表

```sql
-- 应有 1 条日志记录
SELECT log_id, master_id, operation_type, operation_content, operator
FROM pix_outp_reg_master_log
WHERE master_id = (SELECT master_id FROM pix_outp_reg_master
                   WHERE outp_date = '2026-06-14' AND special_clinic_code = '22222'
                   ORDER BY master_id DESC FETCH FIRST 1 ROWS ONLY);
```

**验证点**：
- `log_id` 为 Mini 雪花 ID（≤12 位）
- `operation_content` 包含新增号源信息
