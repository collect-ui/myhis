# Webshell 开发配置踩坑

本文基于本次 `workspace_project + workspace_file` 改造做复盘，聚焦「为什么会出现：有项目、目录却不加载/无法新增」这类问题，以及稳定写法。

## 本次改造落点

- 后端目录模块：`collect/webshell/workspace_file/index.yml`
- 后端路由挂载：`collect/webshell/service.yml`
- 前端页面编排：`collect/frontend/page_data/data/server/webshell.json`
- 新目录表与模型：`sql/webshell_workspace_file.sql`、`model/devops/webshell_workspace_file.go`

## 关键链路（正确认知）

- 项目管理是 `workspace_project_*`，目录管理是 `workspace_file_*`，两套状态和接口要分离。
- 目录树来源是后端 `workspace_file_tree_query`，不是页面内置 mock。
- 目录列表 CRUD 来源是 `workspace_file_query/add/update/delete`。
- `project_code` 是目录链路的唯一业务键，必须贯穿请求与状态。

## 高频踩坑与根因

### 1) 有项目但目录树不请求

现象：切换项目后看不到 `workspace_file_tree_query` 请求。

根因：把树请求放在 tab 内层 `initAction`，它更像初始化触发，不等于每次切换都触发。

建议：

- 项目切换动作里直接触发树请求，且带 `project_code`。
- 若做性能优化，配 `workspaceTreeLoadedMap` 做“首次加载后缓存”。

## 2) 目录管理弹窗显示“暂无数据”

现象：弹窗打开了，但列表一直空。

根因：列表 reload 仅依赖 `appendFormFields`，而弹窗首次打开时，表单可能尚未完成挂载/赋值，导致 `project_code` 丢失。

建议：

- `reload-workspace-file-manage` 直接从 store 取参数，避免强依赖表单生命周期。
- 例如：`project_code` 取 `workspaceFileCurrentProject.project_code || workspaceCurrent`。

## 3) 点击“新增目录”没弹框

现象：按钮点击后无可见变化。

根因：目录弹框定义在“项目管理对话层”里；如果该层未打开，内部弹框即便 `open=true` 也不可见。

建议：

- 短期：触发新增目录时同时确保外层 `workspaceManageVisible=true`。
- 长期：把目录弹框提升到页面根层，解除对项目管理弹窗的层级耦合。

## 4) 新增/编辑目录失败（提示项目编码为空）

现象：接口有调用，但返回 `项目编码不能为空`。

根因：仅使用表单里的 `project_code`，而 disabled 字段或弹窗初始化顺序导致值未可靠传入。

建议：

- 提交请求时强制覆盖：`project_code = workspaceFileCurrentProject.project_code || workspaceCurrent`。

## 5) 目录切换每次都重复请求

现象：切换到已看过的项目还会反复请求树。

根因：无缓存标识，或缓存 key 与项目 key 不一致。

建议：

- 增加：`workspaceTreeMap`（`project_code -> tree`）与 `workspaceTreeLoadedMap`。
- 切换时先读缓存，未加载才请求。
- key 统一使用 `project_code`，不要混用 `value/path`。

## 后端配置踩坑

## 1) `model_save/model_update` 可用但未注册模型

现象：接口配置看起来没问题，但保存/更新失败。

根因：漏了 `model/devops/add_table.go` 的注册。

建议：新增模型后立即注册表名与主键映射。

## 2) 重复路径校验只做前端不够

现象：并发或绕过页面时可写入重复路径。

建议：

- 服务层做重复校验（`workspace_file_query + path_exact`）。
- 数据库层加唯一索引（`project_code, path`）双保险。

## 3) 删除目录未拦截子节点

现象：父目录被删后，子节点悬空。

建议：删除前检查 `parent_id=file_id` 的子节点，非空则拒绝删除并提示。

## 本次可复用的稳定写法

- 状态：
  - `workspaceFileCurrentProject`
  - `workspaceFileSearchForm`
  - `workspaceTreeMap`
  - `workspaceTreeLoadedMap`
- 请求参数兜底：
  - `project_code: workspaceFileCurrentProject.project_code || workspaceCurrent`
- 操作顺序：
  - 打开管理/新增 -> 先写 store -> 再更新表单 -> 再 reload

## 多层 Store 正确姿势（本次最终结论）

- `workspaceFileDialogVisible` 由弹框所在层维护，关闭动作应写本层。
- 从 panel 子层打开弹框：
  - `targetStore: "__parent__"`
  - 同时上抛 `activeStore: ${__this__}`。
- 新增/编辑提交后应做两类刷新：
  - 目录管理列表：在管理层执行 `reload-workspace-file-manage`。
  - 当前 panel 目录树：`targetStore: ${activeStore}` 执行 panel 的树刷新组。
- `reload-init-action` 只有在目标 store 上存在对应 `group` 才会生效；否则是“执行成功但无效果”。

## project_code 统一规则（已验证）

- 工作空间相关 key 统一使用 `project_code`：
  - tabs key、项目切换、目录查询参数、panel 层 `project_code`。
- 不再混用 `project_id/value/path`，否则会出现：
  - panel id 与项目键不一致
  - 缓存命中失败
  - reload 目标错位

## 联调检查清单

1. 切换项目时，首次出现 `workspace_file_tree_query` 请求，再次切回不重复请求。
2. 点“目录管理”时，`workspace_file_query` 请求里一定带 `project_code`。
3. 点“新增目录”能弹窗，提交后列表刷新可见。
4. 重复 `path` 被拦截，错误信息明确。
5. 非空目录删除被拦截。

## 结论

这次问题本质不是接口缺失，而是「低代码 action 执行时序 + 弹窗层级 + 参数来源」三个点叠加。后续只要坚持：

- 业务键统一（`project_code`）
- 请求参数用 store 兜底
- 弹窗避免跨层耦合

就能显著降低 webshell 页面配置回归风险。

## 本轮补充（目录树右键）

1. `tree-select` 下拉树不要直接引用 `workspaceTree` 原对象；应在打开弹窗或树刷新后写入独立变量（如 `workspaceFileParentTree`），并使用新数组引用（如 `[...workspaceTree]`），否则会出现“首次显示 ID、再次打开才显示名称”的旧数据问题。
2. 删除后刷新目录树要在当前 panel 层执行，不能误用上层 store。做法是先记录 `activeStore: ${__this__}`，再用 `targetStore: ${activeStore}` 执行 `reload-workspace-file-tree`，否则会出现“删除成功但树不刷新”。

## 本轮补充（表单与删除）

1. 编辑表单首次空白多为时序问题：`update-form` 早于表单实例挂载。可用“延迟 update-form”而非改 `initialValues` 规避。
2. 表单未展示字段不会自动提交：主键 `file_id` 等要在保存 `ajax.data` 单独补齐，建议来源 `currentRow` 缓存。
3. 目录删除从“逐层删除”改为“批量软删子树”后，确认文案必须同步提示：删除当前目录会连同所有子目录和文件一起删除。
4. 执行 SQL 更新时注意空值渲染：例如 `modify_user` 为空会生成 `modify_user =` 导致语法错，需用模板兜底（`{{if .modify_user}}{{.modify_user}}{{else}}''{{end}}`）。

## 本轮补充（搜索栏与布局）

1. `topRight` 在可拖拽 panel 下可能出现内容裁切，尤其是“按钮+输入框”组合；建议迁移到 `searchToolBar`。
2. 搜索框若写固定宽度，拖窄后容易被遮挡；应使用 `flex1 + minWidth:0 + input width:100%` 自适应。
3. 搜索交互统一（服务器列表、目录树）：`allowClear + SearchOutlined 后缀图标`，减少用户认知切换。
