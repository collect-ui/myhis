# 项目管理配置实战（给大模型/新同学的快速接入版）

本文是 webshell「项目管理」低代码改造的落地手册，目标是让下一次会话能在 3-5 分钟内看懂现状并继续配置。

## TL;DR

- 主文件：`collect/frontend/page_data/data/server/webshell.json`
- 主标识：`project_code`（统一作为 `project_id/value/path`）
- 布尔字段：`show_home` 使用字符串 `"1"` / `"0"`
- 服务器筛选与编辑都用 `tree-select`（数据源 `server_env_data`）
- 用户联动触发点：服务器控件自身 `action`，不是整表单 `submitOnChange`
- 项目管理入口：`topRight` 放操作按钮，`searchToolBar` 放筛选项

## 1. 本次改造目标与结果

从“项目名称+标识简易输入”升级为“可维护的项目表格管理”：

- 支持搜索、分页、增改删
- 支持服务器与用户联动
- 支持目录与 Git 地址维护
- 支持首页展示状态（图标化展示）

## 2. 配置落点（只看这几段）

在 `webshell.json` 重点看 4 段：

1. `initStore`：项目列表/搜索态/弹窗态/表单态
2. “项目管理”按钮：打开管理弹窗
3. “项目管理”对话框：`layout-fit + searchToolBar + topRight + table + pagination`
4. “新增/编辑项目”对话框：表单字段 + 服务器/用户联动

建议定位关键词：

- `workspaceManageVisible`
- `workspaceProjectList`
- `workspace-project-search-form`
- `workspace-project-form`

## 3. 关键数据约定（必须统一）

### 3.1 项目数据模型

- `project_name`：项目名称
- `project_code`：项目编码（唯一标识）
- `order_id`：排序号
- `show_home`：`"1"` 显示，`"0"` 不显示
- `server_id` / `server_name`：服务器
- `server_os_users_id` / `server_user_name`：登录用户
- `project_dir`：项目目录
- `git_repo_url`：Git 地址

### 3.2 搜索模型

`workspaceProjectSearchForm` 建议独立维护：

- `page`、`size`
- `project_name`
- `project_code`
- `server_id`（`tree-select`）
- `git_keyword`

## 4. 交互设计规范（避免后续返工）

### 4.1 搜索区与操作区分离

- `searchToolBar`：只放筛选字段 + 搜索/重置
- `topRight`：只放“新增项目”等操作按钮

### 4.2 服务器选择后自动拉用户

推荐 action 链路：

1. 清空 `workspaceUserList`
2. 清空 `server_os_users_id`
3. 调 `project.server_os_users_query`
4. 有数据时默认选第一项

### 4.3 不使用表单级自动提交

- 不用 `form.submitOnChange` 做联动
- 使用控件级 `action` 更可控、可读性更高

## 5. 组件实现依赖（重要）

`tree-select` 要支持 `action`，否则配置写了也不会触发。

- 组件文件：`F:\collect-ui\src\components\tree\tree-select.tsx`
- 对齐参考：`F:\collect-ui\src\components\select\select.tsx`
- 判定标准：`onChange` 中应调用 `handlerActions(...)`

## 6. 常见坑与快速修复

### 6.1 目录树突然空白

现象：工作空间 `tree` 不加载。

排查：是否把 `workspaceCurrent/path` 改为 `project_code` 后，忘记同步 `workspaceDataMap` 键。

修复：`workspaceDataMap` 键与 `project_code` 一致。

### 6.2 服务器变更后用户不刷新

排查：

1. `tree-select` 是否已支持 `action`
2. `action` 是否挂在服务器控件本身
3. 请求入参是否取 `getFormValue('workspace-project-form','server_id')`

### 6.3 分页总数不对

排查：`pagination.total` 是否用“过滤后数量”而不是全量 `workspaceProjectList.length`。

## 7. UI 可读性建议（当前已实践）

- 服务器、项目编码、用户、排序号、首页展示列给足宽度
- 首页展示用 icon（不再用“是/否”）
- 弹窗内表格高度控制在中等（如 `420`）

## 8. 后端接口接入路线（下一阶段）

建议按顺序替换，逐步从 mock/store 迁移到真实接口：

1. 列表查询（含搜索分页）
2. 新增/编辑/删除
3. 按 `project_id` 加载工作区 tree/fileMap

这样可以保证前端交互先稳定，再做后端替换，联调风险最低。
