# Webshell Editor Pool 源码内容搜索设计方案

## 1. 背景与目标

当前 `webshell-editor-pool` 左侧“源码目录”只支持按文件名/路径关键字过滤树节点（`workspace_file_query.keyword`），不支持按文件内容检索。  
本需求目标是在保持现有交互稳定的前提下，新增“内容搜索”能力：

1. 输入内容关键字，返回命中文件列表。
2. 结果可预览命中行上下文（类似 Sublime 的结果视图）。
3. 点击结果路径可直接打开对应文件。

## 2. 需求范围

### 2.1 In Scope

1. 新增内容搜索入口（独立对话框或独立结果面板）。
2. 返回结果包含：文件路径、行号、命中行、前后文。
3. 点击结果项复用现有逻辑打开文件。
4. 增加异常提示和空结果提示。

### 2.2 Out of Scope（本期不做）

1. 编辑器内自动跳转并高亮到命中行（可在二期加）。
2. 正则高级搜索/替换。
3. 跨项目并行搜索。

## 3. 现状链路分析

### 3.1 前端（低代码）

1. 左侧源码树位于 `collect/frontend/page_data/data/server/webshell_editor_pool_source_fragment.json`。
2. 当前搜索表单 `workspace-tree-filter-form` 仅维护 `keyword`，触发 `reload-workspace-file-tree`。
3. 树数据由 `workspace_file_query` 返回 `name/path` 过滤结果并组树。

### 3.2 后端

1. 目录树接口：`webshell.workspace_file_query`（`collect/webshell/workspace_file/index.yml`）。
2. 文件内容读取接口：`webshell.workspace_file_content`（单文件）/`workspace_file_content_batch`（批量）。
3. 目前没有“按内容检索”服务。

## 4. 方案选型

## 4.1 候选方案

1. 基于 `workspace_file_content_batch` 拉取内容后前端筛选。  
优点：无 SSH shell 依赖。缺点：大项目性能差、网络负载高。

2. 后端通过 SSH + SFTP 按“工作区文件清单”逐文件读取并检索，返回结构化结果。  
优点：不依赖远端 `rg/grep` 环境差异，可直接复用现有项目文件元数据。缺点：实现复杂度略高于 shell 方案。

## 4.2 结论

采用方案 2：新增后端模块 `workspace_content_search`（SFTP 扫描实现），前端只负责输入参数与结果展示。

## 5. 交互设计

## 5.1 入口

在“源码目录”顶部工具条新增内容搜索入口按钮（图标按钮，避免占空间）。

## 5.2 展示形态

采用“左侧源码区内的独立结果面板（可折叠）”：

1. 上半区仍是目录树。
2. 下半区显示内容搜索表单与结果列表。
3. 结果面板默认收起，触发搜索后自动展开。

## 5.3 表单字段

1. `keyword`：关键字（必填，最少 2 字）。
2. `match_case`：是否区分大小写（默认否）。
3. `max_results`：最大返回条数（默认 100，允许 50/100/200）。
4. `include_glob`：可选文件匹配（如 `*.py,*.sql`）。

## 5.4 结果项

每条结果展示：

1. 文件名 + 相对路径。
2. 行号。
3. `before_text` / `match_text` / `after_text` 三段上下文。

点击结果项后执行与“最近文件”一致的打开链路，打开该路径文件。

## 6. 技术设计

## 6.1 后端新增模块（Go）

新增文件：

1. `plugins/module_workspace_content_search.go`
2. `plugins/a_register.go`（注册模块）

模块名：`workspace_content_search`  
服务 key：`webshell.workspace_file_content_search`

建议请求参数：

1. `project_code`（必填）
2. `keyword`（必填）
3. `match_case`（默认 false）
4. `max_results`（默认 100，上限 200）
5. `include_glob`（可选）
6. `context_lines`（默认 1，上限 2）

核心处理流程：

1. 校验参数（关键字长度、上限约束）。
2. 查询当前有效项目（`webshell_workspace_project` 最新记录）。
3. 查询对应服务器用户并解密密码（复用现有解密逻辑）。
4. 建立 SSH 连接。
5. 通过 `webshell_workspace_file` 取文件列表，按路径通过 SFTP 读取文本内容并执行关键字匹配。
6. 提取命中行及上下文，统一转为结构化 `items`。
7. 返回 `summary + items`。

返回结构建议：

```json
{
  "summary": {
    "engine": "sftp_scan",
    "duration_ms": 128,
    "hit_files": 3,
    "hit_count": 8,
    "truncated": false
  },
  "items": [
    {
      "file_path": "/data/project/x/a.py",
      "file_name": "a.py",
      "line_no": 18,
      "before_text": "def foo():",
      "match_text": "    return request.args.get('id')",
      "after_text": ""
    }
  ]
}
```

### 安全与性能约束（SFTP 方案）

1. 不拼接 shell 命令，避免命令注入面风险。
2. 仅在工作区已登记文件范围内检索，不做任意路径扫描。
3. 限制 `max_results/max_scan_files/max_file_bytes/max_matches_per_file` 上限。
4. SSH/SFTP 连接设置超时（当前 15s），请求侧有整体超时控制。
5. 二进制/超大文件自动跳过，降低无效扫描成本。

## 6.2 低代码页面改造

主要文件：

1. `collect/frontend/page_data/data/server/webshell_editor_pool_route.json`
2. `collect/frontend/page_data/data/server/webshell_editor_pool_workspace_fragment.json`
3. `collect/frontend/page_data/data/server/webshell_editor_pool_source_fragment.json`

新增 store 字段（示例）：

1. `workspaceContentSearchVisible`
2. `workspaceContentSearchLoading`
3. `workspaceContentSearchForm`
4. `workspaceContentSearchResults`
5. `workspaceContentSearchSummary`
6. `workspaceContentSearchError`

新增 action group：

1. `reload-workspace-file-content-search`

结果点击动作：

1. 复用现有“树节点/最近文件打开文件”的 method 链路；
2. 将 `row.file_path` 写入 `workspaceActivePath/workspaceCurrentFilePath`；
3. 打开或激活对应 tab。

## 7. 异常与边界

1. 未选择项目：提示“请先选择项目”。
2. 关键字长度不足：提示“至少 2 个字符”。
3. 无结果：显示空态，不报错。
4. SSH/权限异常：显示后端错误文本。
5. 结果被截断：在 summary 标记 `truncated=true` 并提示“仅展示前 N 条”。

## 8. 测试方案

## 8.1 后端验证

1. `go test ./...`（编译回归）。
2. 服务参数校验（空项目、空关键字、超上限）。
3. 有命中/无命中/异常命中路径。

## 8.2 前端与页面链路（无头浏览器）

1. 打开 `#/collect-ui/framework/webshell-editor-pool`。
2. 输入关键字触发内容搜索，验证结果列表渲染。
3. 校验结果项显示路径、行号、上下文。
4. 点击结果项后文件 tab 打开正确路径。
5. 验证空结果与错误提示表现。

## 8.3 回归点

1. 原“文件名过滤树搜索”不回归。
2. 原“最近文件列表”不回归。
3. 原“HTTP 控制台”链路不受影响。

## 9. 实施步骤（建议）

1. 实施后端模块与 service（先打通 JSON 返回）。
2. 接入低代码内容搜索面板与结果渲染。
3. 复用打开文件动作并联调。
4. 无头浏览器全流程回归并产出截图/日志。
