# 菜单路由与数据库配置

这份文档只说明当前项目里“纯后端返回菜单配置”的真实链路，避免再把 `sport-ui` 本地静态文件和 `/data/project/sport/collect` 里的后端页面配置混在一起。

## 1. 真正生效的链路

当前菜单不是前端本地拼出来的，而是这条链路：

1. 前端请求 `post:/template_data/data?service=system.menu_query`
2. 后端服务定义在：
   - `/data/project/sport/collect/system/menu/index.yml`
   - `/data/project/sport/collect/system/menu/menu_query.sql`
3. `menu_query.sql` 从 `sys_menu` 表读取菜单，再按 `parent_id` 转成树
4. 前端 `menu2router` 用菜单记录里的 `url`、`api`、`router_group` 等字段生成路由
5. 页面配置由菜单记录里的 `api` 指向 `frontend.xxx` 服务返回
6. `frontend.xxx` 服务定义在：
   - `/data/project/sport/collect/frontend/page_data/index.yml`
7. `frontend.xxx` 再通过 `data_file` 返回具体 JSON 页面

结论：

- 菜单要生效，核心不是改 `sport-ui/data/menu/menu.json`
- 应该改数据库表 `sys_menu`
- 页面路由对应的 JSON 应该放在 `/data/project/sport/collect/frontend/page_data/data/...`

## 2. 菜单相关表结构

运行配置见：

- `/data/project/sport/conf/application.properties`

当前服务使用：

- `driverName=sqlite3`
- `dataSourceName=./database/price.db`
- `project=base`

所以当前菜单记录实际落在：

- `/data/project/sport/database/price.db`

相关表：

### `sys_menu`

主要字段：

- `sys_menu_id`: 菜单主键
- `menu_type`: `1` 表示目录，`2` 表示可路由菜单
- `menu_name`: 菜单名称
- `menu_code`: 菜单编码，要求唯一
- `parent_id`: 父菜单 ID
- `router_group`: 路由分组编码，当前框架页一般填 `framework`
- `api`: 页面配置接口，真正决定打开哪个后端页面
- `url`: 前端路由路径
- `in_menu`: 是否显示在菜单中，通常填 `1`
- `is_common`: 是否公共菜单，填 `1` 时无需额外角色授权也能显示
- `order_index`: 排序
- `belong_project`: 当前项目，当前环境是 `base`

### `role_menu`

角色菜单授权表：

- `role_menu_id`
- `role_id`
- `sys_menu_id`
- `belong_project`

如果菜单记录 `is_common='1'`，通常可以先不写 `role_menu`。

## 3. 菜单查询规则

菜单查询 SQL 在：

- `/data/project/sport/collect/system/menu/menu_query.sql`

几个关键点：

- `where a.belong_project = {{.belong_project}}`
- `router_group` 会通过 `left join sys_menu sm on a.router_group = sm.menu_code` 回查框架组的 `group_path` 和 `group_api`
- `with_role=true` 时，会根据 `is_common='1'` 或 `role_menu` 做可见性过滤

这意味着新增一个框架子页面时：

- 不需要自己写 `group_path` / `group_api`
- 只要把 `router_group` 正确填成现有框架组编码即可

当前框架组编码是：

- `framework`

当前“运维管理”目录菜单是：

- `menu_code=devops`
- `sys_menu_id=991e3ec0-df57-4e15-80da-1f895e533e30`

## 4. 新增一个菜单路由的最小步骤

以 `Editor工作台` 为例，正确步骤是：

1. 在 `/data/project/sport/collect/frontend/page_data/index.yml` 新增一个 `frontend.xxx` 服务
2. 给这个服务绑定一个 `data_file`
3. 在 `/data/project/sport/collect/frontend/page_data/data/server/` 下新增对应 JSON 页面
4. 往 `sys_menu` 插入一条 `menu_type='2'` 的子菜单
5. `api` 指向 `post:/template_data/data?service=frontend.xxx`
6. `url` 指向前端路由，例如 `/framework/webshell-editor-pool`
7. `router_group='framework'`
8. `parent_id` 指向目标目录菜单，例如 `devops`

## 6. 排障规则

如果“菜单看不到”，按下面顺序查：

1. 查 `sys_menu` 是否真有这条记录
2. 查 `belong_project` 是否等于当前配置里的 `project`
3. 查 `parent_id` 是否挂到正确目录
4. 查 `in_menu` 是否为 `1`
5. 查 `is_common` 是否为 `1`
6. 如果不是公共菜单，再查 `role_menu`
7. 查 `api` 指向的 `frontend.xxx` 是否在 `page_data/index.yml` 中定义
8. 查 `data_file` 是否存在
9. 查 `url` 是否和预期路由一致
10. 查 `router_group` 是否是现有框架组编码

如果“菜单能看到但点不开”，优先查：

1. `api`
2. `frontend.page_data/index.yml`
3. `data_file`
4. 页面 JSON 语法

## 7. 不要再走错的地方

下面这些不是当前这套正式菜单的主配置入口：

- `/data/project/sport-ui/data/menu/menu.json`
- `/data/project/sport-ui/data/app.json` 里手工拼菜单
- 仅新增一个前端本地 html 入口

这些最多只能做本地调试，不是当前生产/测试链路里 `system.menu_query` 的真实来源。
