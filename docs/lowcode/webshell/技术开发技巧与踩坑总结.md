# WebShell 技术开发技巧与踩坑总结

本文汇总本轮 WebShell 项目管理/工作区文件管理改造中的关键经验，便于后续快速复用。

## 1. 目录扫描与同步

### 1.1 Shell 扫描比 SFTP 递归更适合全量树
- 场景：目录很大（如 `node_modules`）时，SFTP 多次 `ReadDir` RTT 高，整体慢。
- 实践：优先用 SSH Shell 一次执行 `find`，再在 Go 侧解析树。
- 经验：
  - 必须支持排除目录（`exclude_dirs`），优先剪枝。
  - 需要兼容不同输出格式（分隔符失效、粘连输出）。
  - 对 `find` 的非 0 退出码要容错，但不能吞掉“路径不存在/权限不足”等关键错误。

### 1.2 `find` 输出解析要做多层兼容
- 已踩坑：`	` 分隔在部分环境失效，导致形如 `d4096.../path` 粘连。
- 推荐策略：
  1) 优先按显式记录前缀（如 `__REC__`）切段；
  2) 再按主分隔符（如 `|`）；
  3) 最后紧凑格式兜底解析。

### 1.3 根路径要“解析后再确认”
- 已踩坑：传入 rootPath 与实际扫描路径规范不一致（相对/绝对）会导致树挂载为空。
- 经验：先根据结果集推导 `resolvedRoot`，再做 parent-child 关联。

## 2. 增量同步（替代全量删插）

### 2.1 增量比对键应优先 `path`
- 结论：文件树同步以 `project_code + path` 为稳定业务键。
- 避免：用临时 UUID 作为差异比对主键，会导致每次都判定为新增。

### 2.2 `get_modify_data` 输出结构要先验证
- 已踩坑：误用 `.item.after.xxx` 导致 `add_file_list` 为空。
- 经验：先打印一条 `change_list` 样本，确认字段在顶层还是 `before/after/copy`，再写模板。

### 2.3 增量链路推荐
- 查询远端 `file_list`
- 查询本地 `local_file_list`
- `get_modify_data` 产出 change
- 拆分 `add/remove`
- 新增批量写入，删除物理删除（按 file_id 列表）

## 3. 前端树与搜索性能

### 3.1 懒加载优于全量树渲染
- 改造点：初始化只拉根节点（`root_only=true`），展开按 `parent_id` 请求子节点。
- 收益：避免大数据量一次性塞前端导致卡顿。

### 3.2 搜索触发策略
- 已踩坑：`submitOnChange` + 单字符搜索会造成高频查询卡顿。
- 方案：
  - `submitOnEnter: true`
  - 关键词长度阈值（建议 >=2）

### 3.3 搜索结果树构建
- 已踩坑：重复节点、空目录、路径展示混乱。
- 方案：
  - 后端返回 `path_display`（去项目根前缀）
  - 前端按 `path_display` 组树
  - 按 `path` 去重
  - 仅以文件结果建树并 prune 空目录

## 4. 低代码渲染层常见坑

### 4.1 `service2field` 不适合复杂条件模板
- 已踩坑：`exclude_dirs: "{{if ...}}"` 在 service 参数中未按预期生效。
- 方案：先用 `update_field` 计算中间值，再传 `[exclude_dirs_effective]`。

### 4.2 `listview` 正确用法
- 已踩坑：使用 `itemRender/itemStyle/emptyText`（不符合当前实现）导致运行报错。
- 当前实现（`collect-ui`）应使用：
  - `itemData`
  - `itemAttr`
  - `keyField`
  - `rowClickAction`

### 4.3 作用域差异（`row` 在 adapt 中不可用）
- 已踩坑：`ajax.adapt` 里直接用 `row.path` 报 `Error in functionString`。
- 方案：在前置 `update-store` 先写入路径，再在 adapt 用 store 字段。

### 4.4 按钮冒泡与行点击冲突
- 现象：点击删除按钮触发行点击（打开文件），删除看起来“没反应”。
- 方案：按钮支持 `stopPropagation: true`，并在组件层实现 `e.stopPropagation()`。

## 5. 编辑器语言高亮

### 5.1 不要在页面层硬编码长后缀判断
- 方案：统一在 editor 组件内做 `inferLanguageFromPath`，页面仅传空或显式 language。

### 5.2 Vue 高亮现实
- 已踩坑：手写简化 `vue` tokenizer 只能高亮 template，script 效果差。
- 方案：先将 `.vue` 映射到 `html` 作为实用兜底；若需 IDE 级效果需接入 Vue 语言服务（Volar 类方案）。

## 6. 最近常用文件

### 6.1 后端建模建议
- 表：`webshell_workspace_file_recent`
- 唯一键：`(project_code, file_path)`
- 字段：`open_count`, `last_open_time` 等
- 排序：`open_count DESC, last_open_time DESC`，固定最近 20 条

### 6.2 点击记录与删除
- 打开文件（目录树/最近列表）都要调用 `workspace_file_recent_open`
- 最近列表删除用 `workspace_file_recent_delete`

## 7. 建议的排查顺序（实战）

1. 先看网络请求是否发出（enable 条件是否命中）
2. 再看返回结构（`data` 是字符串/对象/嵌套）
3. 再看 store 写入路径是否正确（key/path）
4. 最后看渲染层组件契约（listview/button/tabs）

## 8. 大页面拆分为两个低代码变量（Webshell 实战）

### 8.1 标准拆分方式
- 在主服务（如 `frontend.webshell`）里通过 `service2field` 注入两个变量：
  - `webshell_ssh_fragment`
  - `webshell_editor_fragment`
- 两个片段服务使用 `module: empty + file2datajson + param2result`，分别读取同目录 JSON 文件。
- 主页面通过 `{{to_json .变量名}}` 或 `{{to_json .变量名.子段}}` 渲染片段。

### 8.2 推荐边界
- SSH 变量：登录/执行主链路 + 相关弹窗 + 浮动入口。
- Editor 变量：整个 drawer（源码目录、HTTP 目录、右侧面板等都放在 drawer 片段里）。
- 经验：按“功能模块边界”拆，不要按“单个深层 children”拆。

### 8.3 最终收益
- 主 `webshell.json` 体积显著下降，骨架更清晰。
- 可单独读取某个片段文件定位问题，避免每次在巨型 JSON 内检索。
- 变更影响面可控：SSH 与 Editor 各自演进，冲突更少。

### 8.4 防坑清单
- 片段内禁止再引用自身 `to_json`（避免递归）。
- `service2field` 注入顺序应在主 `file2datajson` 之前。
- 每次拆分后都要做 JSON/YAML 语法校验 + 关键交互回归。

### 8.5 Tree 标题展示推荐（函数变量方式）
- 对 `tree` 组件，优先在后端 SQL 或前置 adapt 中生成展示变量（如 `display_title`）。
- 页面层只写简单绑定：`treeTitle: "${item.display_title||item.title}"`。
- 避免把复杂拼接逻辑直接堆在 `treeTitle`（排障困难，且不同上下文不易复用）。

---

如后续扩展：建议优先补“统一调试字段（count/raw_len/resolved_root）”与“错误可视化”，可显著降低线上定位成本。
